## 3. 청중 가정 (v8 구성안에서 가져옴)

평가자·포트폴리오 보는 사람 + 데모데이·잠재 도입자 (병행).

**청중이 3분 후 가져갈 한 줄**:

이 한 줄을 만드는 것이 발표의 전부.

## 4. 의사결정 로그

### 4-1. 메인 웹사이트 — making-of vs showcase

| | making-of (Next.js, 7-Act) | showcase (HTML, 10섹션) |
|---|---|---|
| 정체성 | 팀×AI 협업 증거 사이트 | 하루안부 제품 자체 소개 |
| 주체 | 우리 (AI는 매체) | 제품·사용자 |
| 채택 여부 | ❌ | ✅ |

**→ showcase 채택.**
**왜:** 3분 발표의 목적이 "제품 소개"이고 청중이 평가자/도입자이기 때문. making-of는 별도 발표(또는 사이트 자체 둘러보기)용으로 남김. 두 사이트는 정체성이 의도적으로 분리돼 있으므로 한 발표에 섞으면 메시지가 흐려짐.

### 4-2. Claude orb 등장 여부

처음엔 "showcase 발표라면 orb는 빼는 게 사이트 framing과 정합"이라 권유했으나, **Shon이 orb 유지를 결정**.

**채택된 framing**: orb는 *외부의 공동 발표자*가 아니라 *제품 안에서 일하는 AI* (사이트 푸터의 "POWERED BY CLAUDE", SOS Opening ④단계의 "AI 판단 (Claude 분석 패널)"과 동일 인격).

이렇게 정체성을 묶으면:
- 사이트 정체성과 발표가 일관
- "공동 발표자" 어색함 회피
- "AI가 진짜 제품 안에서 일한다"를 *시각(사이트)* + *청각(orb 음성)* 두 채널로 박을 수 있음

### 4-3. Orb 발화 비율

| 후보 | 채택 여부 |
|------|---------|
| Shon 80% / Orb 20% (orb는 2번만 등장) | ❌ (초기안) |
| Shon 50% / Orb 50% | ❌ |
| **Shon 20% / Orb 80%** | ✅ (최종) |

**→ Shon이 "orb가 최대한 대화를 이끌어가는 형식"으로 결정.**
**유의점:** orb 비중이 너무 크면 청중이 "발표"가 아닌 "비디오 시청"으로 인식할 위험. Shon의 짧은 인간 터치(인트로·SOS 받아침·Demo·송별)를 핵심 자리마다 배치해서 무대의 따뜻함 유지.

### 4-4. Orb 호출 방식 — 자연스러운 호흡

| 후보 | 채택 여부 |
|------|---------|
| A. 명시적 호출 ("Claude, 그때 어땠지?") | ❌ Siri 클리셰 |
| **B. 회상 트리거 ("이 화면 만들 때 진짜 고민했던 게...")** | ✅ (선택) |
| C. 자막 인터뷰 (화면에 Q 띄우기) | ❌ |
| D. 게스트 모시기 (인사 멘트) | ❌ 시간 잡아먹음 |

**→ B 채택 (자연스러운 호흡).**
구체 적용: 자체 발표에서는 orb가 메인 화자이고 Shon이 트리거를 짧게 깔아주는 형태로 변형됨. (예: "이 순간 시스템 안에서 무슨 일이 벌어지냐면..." → orb 등장)

### 4-5. Orb 발화 톤

- **1인칭 "저" 일관 사용** (3인칭 "AI가" 금지) — 인격을 부여해서 신뢰감 형성
- **단정문 종결** ("~합니다", "~됩니다") — 질문/권유 금지
- **숫자는 한글로 풀어 읽기** ("0.3초" → "영 점 삼 초", "47명" → "마흔일곱 명") — TTS 발음 정확도 + 청자 인식
- **차분·신뢰감·따뜻함**. 격앙·과장 금지

### 4-6. 발표 시간 — 3분 고정

| 화자 | 단어 수 | 시간 |
|------|---------|------|
| Shon | 약 85단어 | 약 36초 |
| Orb | 약 280단어 | 약 120초 |
| pause/전환 | — | 약 24초 |
| **합계** | **약 365단어** | **180초** |

3분은 짧음. 10섹션 중 시그니처 4모먼트(SOS·Features 5앱·Demo·Epilogue)에 시간 몰아주고 나머지(Pain·Persona·Design Language)는 *스크롤로 흘려보내는* 전략.

### 4-7. Orb 음성 도구 — ElevenLabs

후보를 검토하고 떨어뜨린 기록 (`voice-tests/` 폴더에 실제 샘플 mp3 보관):

| 후보 | 음질 | 비용 | 한국어 | 채택 여부 |
|------|------|------|--------|---------|
| macOS `say -v Yuna` | "ㅈㄴ 별로" (Shon 평가) — 옛날 보이스 | 무료 | ✅ | ❌ |
| Microsoft Edge TTS (SunHi, InJoon, HyunSu) | 한 단계 위, 자연스러움 보통 | 무료 | ✅ | ❌ |
| **ElevenLabs Voice Design** | 인간 수준, 1인칭 페르소나 부여 가능 | 유료 (무료 크레딧 충분) | ✅ | ✅ |

**→ ElevenLabs 채택.**
**왜:** 발표 임팩트를 위해 "이게 정말 AI인가?" 싶을 정도의 자연스러움이 필요. Edge TTS는 평이한 안내 보이스 톤이라 orb 인격 부여가 어려움. ElevenLabs는 Voice Design으로 "from inside the system" 페르소나를 직접 조각 가능.

### 4-7-1. v8 번들로 사이트 이전 (2026-05-25 추가)

발표 컨셉이 확정된 직후 (2026-05-25), Shon이 사이트 빌드를 가속해서 `haru-anbu-showcase-v8-bundle/haru-anbu-showcase-v8.html`로 정본을 옮김. 기존 `wireframe-preview.html`은 `_archive/showcase-legacy/`로 이관됨.

**v8에서 새로 생긴 섹션 (와이어프리뷰 대비)**
- `#problem` — Pain Voices 위치
- `#app-preview` — 앱 미리보기
- `#design` — "기능 이전에, 안심이 보이는 형태"
- `#service-features` — **6개 핵심 기능** (AI 안심지수 / 자동 일일 리포트 / 새벽 SOS 3분 응답 / 이상 패턴 감지 / 세 시선 통합 인계 / 음성 인계 자동 정리)
- `#feature-bridge-visual` — Features 트랜지션
- `#flagship-features` — 플래그십 기능 (AI가 매일 정리 / 기록하는 순간 가족에게 / 한국어 워크플로우 / 데스크에서도 회진 중에도)

**v8에서 위치 이동된 섹션**
- SOS (`#flow-overlay`) — 와이어프리뷰에서는 0번 오프닝이었으나, v8에서는 design 다음으로 이동 (중간 위치). 발표에서는 URL 앵커(`#flow-overlay`)로 점프해서 시그니처 오프닝 자리를 유지.

### 4-7-2. v8의 새 섹션 시그니처 모먼트 채택 여부

| 후보 | 결정 |
|------|------|
| 옵션 A — 5앱 줄이고 6기능 추가 | ❌ |
| 옵션 B — Demo 줄이고 6기능 추가 | ❌ |
| **옵션 C — 6기능 스킵 (스크롤로 지나감)** | ✅ |

**→ 옵션 C 채택.**
**왜:** 기존 시그니처 4모먼트(SOS · 5앱 · Demo · Epilogue)의 임팩트가 입증된 흐름. 6기능 섹션은 사이트 둘러볼 때 청중이 발견하는 자리로 둠. 3분에 새 섹션을 끼우면 기존 모먼트 어딘가가 압축되어 인상이 약해질 위험이 더 큼.

**스크롤로 통과시키는 v8 섹션 (발표 중 멈추지 않음)**
- `#problem`, `#app-preview`, `#design`
- `#service-features`, `#feature-bridge-visual`, `#flagship-features`
- (기존) `#features-by-role` 안의 일부 sub-screen

### 4-8. Orb 음성 청크 분할

| 후보 | 채택 여부 |
|------|---------|
| 전체 3분짜리 한 파일 (자동 재생) | ❌ — 발표 중 클릭 흐름과 동기화 불가 |
| **섹션별 16개 청크 (클릭마다 재생)** | ✅ |

**→ 청크 분할 채택.**
**왜:** 무대 위 흐름이 청중 반응에 따라 미세 조정되어야 하기 때문. 정해진 영상 재생이 아닌 *클릭당 한 청크* 구조여야 발표자가 호흡을 컨트롤할 수 있음.

## 6. 향후 재사용 시 유의사항

이 발표 패키지는 *하루안부 쇼케이스 v8 와이어*에 묶여 있어요. 사이트 와이어가 크게 바뀌면 (예: 섹션 추가/삭제, SOS 단계 변경, Features 앱 순서 변경) 아래 문서들을 함께 업데이트해야 정합성이 유지됩니다.

- `01_발표대본_3분.md` — 화면 큐가 사이트 섹션과 묶임
- `02_orb_음성생성계획.md` — 청크 텍스트가 화면 헤드라인과 묶임
- `03_큐시트.md` — 타임라인이 사이트 흐름과 묶임

또한 청중이 바뀌면 (예: 평가자가 아니라 *돌봄 종사자* 또는 *경영진*이 청중이 되면) 시그니처 4모먼트의 선택도 다시 정해야 해요. SOS Opening은 어느 청중에게나 강력하지만, Impact의 ROI 강조 비중은 도입자·평가자에게는 크고 종사자에게는 작아야 합니다.
